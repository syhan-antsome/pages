📊 회원 관리 테이블 관계도 (최종)

approval_status VALUES:
- PENDING    (승인 대기)
- APPROVED   (승인 완료)
- REJECTED   (승인 거부)
- SUSPENDED  (자격 정지)

2. 관계 설명

members ↔ member_roles (1:N)

  • 관계: @ElementCollection (JPA)
  • 목적: 한 회원이 여러 역할을 가질 수 있음
  • 제약:
    • member_id + role이 복합 PK
    • 같은 회원이 같은 역할을 중복으로 가질 수 없음

members ↔ corporate_info (1:1)

  • 관계: @OneToOne (JPA)
  • 목적: 법인 회원 정보 저장
  • 제약:
    • member_id가 FK이자 UK (unique)
    • business_number도 unique (사업자등록번호 중복 방지)

4. 핵심 특징

✅ 개선된 점

  1. 중복 방지: 전화번호 기준으로 1인 = 1 Member 행
  2. 다중 역할: member_roles 테이블로 N개 역할 관리
  3. 법인 분리: corporate_info 별도 테이블로 법인 정보 관리
  4. 승인 워크플로우: approval_status로 상태 관리

✅ 비즈니스 로직

회원가입 시:
1. phone_number로 기존 회원 검색
2. 존재하면 → 역할만 추가 (member_roles INSERT)
3. 없으면 → 새 회원 생성 + 역할 추가
4. CORPORATE 타입이면 → corporate_info 생성 (PENDING 상태)

5. SQL 쿼리 예시

회원 역할 조회

SELECT 
    m.id, 
    m.name, 
    m.email,
    GROUP_CONCAT(mr.role) as roles
FROM members m
LEFT JOIN member_roles mr ON m.id = mr.member_id
WHERE m.phone_number = '010-9000-0001'
GROUP BY m.id;

법인 회원 조회 (승인 상태 포함)

SELECT 
    m.*, 
    ci.company_name,
    ci.approval_status
FROM members m
JOIN member_roles mr ON m.id = mr.member_id AND mr.role = 'CORPORATE_USER'
LEFT JOIN corporate_info ci ON m.id = ci.member_id
WHERE ci.approval_status = 'PENDING';

이 구조로 1인 다역할 시스템이 완벽하게 구현되었습니다! 🎉
